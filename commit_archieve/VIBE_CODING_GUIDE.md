# MiroFish 바이브코딩 가이드

이 문서는 앞서 정리한 로드맵을 기준으로, 혼자서 `바이브코딩` 방식으로 MiroFish류 제품을 다시 만드는 경우 필요한 준비, 개발 순서, 피드백 루프, AI 프롬프트를 한 번에 볼 수 있게 정리한 문서다.

## 한 줄 전략

`그래프를 완벽하게 만드는 것`부터 시작하지 말고, `업로드 -> 예측 질문 -> 결과 보고서`가 끝까지 도는 가장 얇은 데모를 먼저 만든 뒤 그래프와 시뮬레이션의 정밀도를 올린다.

## 먼저 준비할 것

| 항목 | 준비 내용 | 왜 필요한가 |
| --- | --- | --- |
| 대표 유스케이스 1개 | 예: 뉴스/정책 문서를 넣고 향후 반응을 예측 | 기능이 흩어지지 않게 중심 축을 고정하기 위해 |
| 데모 입력 3개 | seed 문서, 예측 질문, 기대 결과 예시 | 매일 같은 기준으로 품질을 비교하기 위해 |
| 출력 포맷 | 예: `요약`, `핵심 행위자`, `시나리오 3개`, `근거` | 리포트 품질을 흔들리지 않게 하기 위해 |
| 기술 스택 | 예: Vue + FastAPI + SQLite + 파일 저장 + JSON 그래프 | AI가 불필요한 스택 변경을 하지 않게 하기 위해 |
| 평가 기준 | 근거성, 일관성, 속도, 비용, 가독성 | 매일 피드백 기준을 고정하기 위해 |
| 저장 단위 | `project_id`, 입력 문서, 그래프, 로그, 리포트 | 각 단계 결과를 다음 단계 입력으로 연결하기 위해 |

## 2주 개발 순서

| Day | 목표 | 완료 기준 | 피드백 포인트 |
| --- | --- | --- | --- |
| 1 | 가짜 엔드투엔드 데모 | 업로드 -> 질문 입력 -> 결과 화면까지 목업으로 연결 | 유저가 흐름을 1분 안에 이해하는가 |
| 2 | 프로젝트/세션 구조 | `project_id` 생성, 입력 파일 저장, 상태 조회 가능 | 실행 단위가 명확히 분리되는가 |
| 3 | 문서 추출기 | 문서에서 엔터티/관계/시간 정보가 JSON으로 저장됨 | 추출 결과를 사람이 읽고 납득할 수 있는가 |
| 4 | 그래프 UI | 노드/엣지 표시, 상세 패널, 재로드 가능 | 그래프가 결과 해석에 실제 도움이 되는가 |
| 5 | 에이전트 프로필 생성 | 그래프 기반 agent profile 3~10개 생성 | 프로필이 질문과 관련 있는가 |
| 6 | 1라운드 시뮬레이션 | agent 간 상호작용 로그 1회 생성 | 로그가 단순 문장 나열로 끝나지 않는가 |
| 7 | 다중 라운드 시뮬레이션 | 5~10라운드 실행, 상태 누적, 중간 로그 조회 가능 | 시간이 지나며 변화가 보이는가 |
| 8 | 리포트 생성 | 로그를 요약/시나리오/근거 형태로 변환 | 로그 원문 없이도 결과를 이해할 수 있는가 |
| 9 | 후속 대화 | 리포트 기반 Q&A, 특정 agent 재질문 가능 | "왜?" 질문에 근거를 제시하는가 |
| 10 | 히스토리/안정화 | 과거 프로젝트 조회, 재실행, 실패 케이스 방어 | 재실행해도 같은 흐름이 재현되는가 |

## 단계별로 꼭 필요한 피드백

- 입력 예시 3개로 매일 직접 돌려본다.
- 실패 케이스 1개를 일부러 넣어본다.
- 결과를 보고 "이걸 믿고 다음 행동을 정할 수 있나?"를 스스로 묻는다.
- LLM 출력 형식이 조금 흔들려도 파이프라인이 깨지지 않는지 확인한다.
- 응답 시간과 비용을 함께 기록한다.

## 구현 우선순위

| 우선순위 | 먼저 할 것 | 나중에 미룰 것 |
| --- | --- | --- |
| P1 | 업로드 -> 질문 -> 결과의 얇은 연결 | Docker |
| P2 | 그래프 초안과 agent profile | 히스토리 아카이브 |
| P3 | 단일 라운드 -> 다중 라운드 시뮬레이션 | 화려한 UI |
| P4 | 리포트 생성과 후속 대화 | 다국어 README |
| P5 | 히스토리, 배포, 안정화 | 멀티 모델 지원 |

## Codex 구현 프롬프트 템플릿

```text
목표:
현재 repo에 [기능명]을 구현해줘.

배경:
이 프로젝트는 문서 -> 그래프 -> 시뮬레이션 -> 리포트 -> 상호작용 흐름의 예측 제품이다.
이번 작업은 그중 [현재 단계]만 구현한다.

입력:
- 예시 입력 1: ...
- 예시 입력 2: ...

완료 기준:
- [API/화면/저장 구조]가 실제로 동작한다.
- 실패 시 사용자에게 보이는 에러가 있다.
- 최소 1개 검증 방법을 같이 남긴다.

비목표:
- UI 화려하게 만들기
- Docker
- 인증
- 과한 리팩터링

작업 원칙:
- 기존 스택 유지
- 변경은 작은 단위로
- mock으로 먼저 연결하고, 이후 실제 로직으로 교체 가능하게 작성
- 변경 파일과 검증 결과를 마지막에 요약

지금 바로 구현해.
```

## Claude 리뷰 프롬프트 템플릿

```text
다음 기능/변경을 리뷰해줘.

리뷰 우선순위:
1. 실제 버그 가능성
2. 상태 관리/데이터 흐름 문제
3. 프롬프트/LLM 출력 변화에 대한 취약점
4. UX 혼란
5. 누락 테스트

출력 형식:
- Findings: 심각도 순
- Assumptions: 내가 놓친 가정
- Next fixes: 바로 고칠 3개

기능 설명:
[여기에 구현한 기능 설명]

변경 파일:
[파일 목록 또는 diff]
```

## Day별 실행 프롬프트

### Day 1-2

```text
문서 업로드부터 결과 페이지까지 연결되는 최소 데모를 만들어줘.
내부 분석은 mock 데이터여도 되지만 `project_id`, 상태 조회, 결과 페이지 이동은 실제로 동작해야 한다.
```

### Day 3-4

```text
업로드된 텍스트에서 엔터티/관계/시간 정보를 추출해 JSON 그래프로 저장하고, 프론트에서 노드/엣지와 상세 패널을 볼 수 있게 구현해줘.
```

### Day 5

```text
그래프를 입력으로 agent profile을 생성하는 백엔드와 설정 화면을 만들어줘. profile은 이름, 역할, 관점, 기억 요약을 포함해야 한다.
```

### Day 6-7

```text
agent profile을 기반으로 1라운드 시뮬레이션부터 시작해 다중 라운드까지 확장해줘. 각 라운드 로그와 상태 변화가 저장돼야 한다.
```

### Day 8

```text
시뮬레이션 로그를 바탕으로 보고서를 생성해줘. 출력 형식은 `요약`, `핵심 행위자`, `시나리오`, `근거` 4개 섹션으로 고정해줘.
```

### Day 9

```text
생성된 보고서를 컨텍스트로 후속 질문을 받을 수 있는 Q&A 화면을 만들어줘. 특정 agent를 선택해서 다시 질문할 수 있어야 한다.
```

### Day 10

```text
과거 프로젝트 목록, 최근 리포트 조회, 재실행 버튼을 추가하고, 실패하기 쉬운 케이스 5개를 방어하는 안정화 작업을 해줘.
```

## 매일 끝날 때 확인할 질문

- 오늘 만든 출력이 내일 단계의 입력으로 바로 쓰일 수 있는가
- 지금 데이터 구조가 임시방편이라 다음 단계를 막고 있지 않은가
- LLM 출력이 흔들리면 어디서 깨지는가
- 사용자가 지금 단계 결과를 보고 "쓸모 있다"고 느낄까
- 지금 고쳐야 할 것과 나중에 미뤄도 되는 것이 분리되어 있는가

## 실행 원칙 요약

1. 기능을 한 번에 크게 시키지 말고 하루 단위로 잘라서 구현한다.
2. 매일 직접 돌려보는 입력 세트를 고정한다.
3. 실패 원인을 문장으로 기록하고 다음 프롬프트에 넣는다.
4. 그래프의 완성도보다 엔드투엔드 흐름을 먼저 만든다.
5. 마지막에만 히스토리, Docker, 안정화 작업을 붙인다.

## 짧은 결론

바이브코딩으로 이 프로젝트를 다시 만들려면 `그래프 기술` 자체보다 `예측 질문 하나를 처음부터 끝까지 처리하는 최소 제품 흐름`을 먼저 만드는 것이 핵심이다. 그 위에 그래프 정밀도, 시뮬레이션 깊이, 리포트 품질, 상호작용성을 순서대로 쌓는 방식이 가장 안전하다.
